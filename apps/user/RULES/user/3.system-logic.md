# System Logic: Services & Doctors

Internal rules for how data is filtered and presented to users.

## 1. Service Visibility
- **General vs Specialized**: Controlled by the `tier` column in the `services` table.
- **Enrollment Filter**: In the specialized booking flow, only dentists explicitly enrolled in that service (via `dentist_services` junction table) are considered eligible.

## 2. Doctor Rendering Rules
- **Active Status**: Only dentists with `is_active: true` are shown.
- **Shift Validation**: Doctors only appear as "Available" if their daily shift covers the **full duration** of the selected service.
- **Block Protection**: Doctors marked as "On Leave" in the `dentist_availability_blocks` table are excluded from slot calculations for those dates.

## 3. Claiming Logic (Waitlist)
- **Token Security**: Every waitlist offer generates a cryptographically secure `claim_token`.
- **Validation**: When claiming, the system verifies:
    1. The token matches the waitlist entry.
    2. The status is `NOTIFIED`.
    3. The `expires_at` timestamp is in the future.
- **Collision Handling**: Claiming creates a fresh appointment. If the slot was taken by someone else 1 second earlier, the claim fails with a "Slot Taken" error.
